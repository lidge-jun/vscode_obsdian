## Part 1 — Easy Explanation

Phase 8 proved that `.hwp` and `.hwpx` files can render inside VS Code through
`@rhwp/editor`, but the first implementation is not safe enough to ship as an
editable document feature.

The visible symptom is:

```text
Failed to load: Request timeout: loadFile
```

The deeper issue is not file size. The tested file is only about 180 KB and the
document still renders. The problem is the integration shape:

- a live external iframe controls document loading and export;
- the bridge uses permissive `postMessage`;
- VS Code sees the editor as read-only while the UI permits edits and saving;
- save writes can overwrite local `.hwp` siblings without confirmation;
- the WebView has no explicit CSP;
- `@rhwp/editor` has a fixed 10 second request timeout and no late-success
  recovery.

Therefore Phase 8.2 changes the direction from "ship editable HWP immediately"
to "make HWP safe first, then re-enable editing only after VS Code lifecycle
support exists."

## Part 2 — Diff-Level Recovery Plan

### Current Blocker Summary

| # | Finding | Severity | Required outcome |
|---|---------|----------|------------------|
| 1 | External `https://edwardkim.github.io/rhwp/` iframe can influence local save bytes through an unverified `postMessage` bridge | Critical | Remove live remote iframe from default path or enforce strict origin/source/schema validation |
| 2 | Editable rhwp UI runs inside `CustomReadonlyEditorProvider` | Critical | Downgrade to viewer-only or implement real editable `CustomEditorProvider` lifecycle |
| 3 | `loadFile` has fixed 10s timeout and stale error state | High | Replace fatal timeout UX with explicit ready/load events and recoverable state |
| 4 | `.hwpx` converted `.hwp` sibling can overwrite existing files silently | High | Add collision checks, save dialog, and atomic writes |
| 5 | React WebView has no explicit CSP | High | Add strict CSP for bundled scripts, styles, fonts, images, frames, and connect endpoints |
| 6 | file bytes are copied as JSON number arrays | Medium | Add size policy now; use transferable/binary bridge in secure local bridge |
| 7 | event bus has no payload validation | Medium | Namespace HWP messages and validate payloads at both boundaries |
| 8 | npm audit reports high/critical dependency issues | High | Track separately from HWP bridge, but block marketplace release until triaged |

### Non-Negotiable Release Gate

Do not publish a VSIX as "HWP/HWPX editable support" until all of these are
true:

```text
1. No live remote rhwp iframe is used by default.
2. WebView HTML contains a reviewed CSP.
3. HWP/HWPX message bridge validates event source, origin, type, and payload.
4. VS Code save lifecycle is honest:
   - viewer-only means no edit/save affordance;
   - editable means dirty state, Cmd+S, Save All, revert, and backup are wired.
5. HWPX->HWP conversion never overwrites an existing sibling without explicit user confirmation.
6. Manual Insiders QA opens the failing HWPX without stale timeout banner.
7. `npx tsc --noEmit` passes.
8. `npm run build` passes.
9. `npm audit --omit=dev` high/critical findings are fixed or explicitly accepted in a documented dependency-risk phase.
```

### Phase 8.2A — Immediate Safety Downgrade

Goal: make the current feature honest and prevent data-loss/security surprises
before deeper work begins.

#### MODIFY `src/react/view/hwp/Hwp.tsx`

Planned behavior:

```diff
- Show editable rhwp controls as if HWP editing is supported.
- Always show "Save HWP".
+ Treat Phase 8.2A as viewer-first.
+ Hide or disable "Save HWP" by default.
+ Show a compact warning banner:
+   "HWP/HWPX editing is disabled until the secure local bridge is installed."
+ Keep rendering if load succeeds.
+ If `loadFile` times out, show a non-blocking warning and stop the spinner.
+ Late success detection is deferred to 8.2C `onLoadStatus` bridge callback wiring.
```

Implementation notes:

- This is a temporary safety move, not the final product direction.
- If `@rhwp/editor` cannot provide a reliable loaded event, do not pretend the
  timeout is fixed. Display a non-blocking warning and keep the document visible.
- Save must remain unavailable until Phase 8.2C/8.2D.

#### MODIFY `src/provider/handlers/hwpHandler.ts`

Planned behavior:

```diff
- Register `saveHwp` as an always-active write path.
+ Either remove the `saveHwp` listener in viewer-only mode,
+ or guard it behind an explicit experimental setting that defaults false.
```

Required setting if an escape hatch is kept:

```json
"vscode-obsdian.hwp.experimentalSave": {
  "type": "boolean",
  "default": false,
  "description": "Enable experimental HWP export/save. Disabled by default because HWP editing lifecycle is not yet integrated with VS Code save semantics."
}
```

### Phase 8.2C — Secure Bridge And CSP

Goal: make WebView boundaries explicit.

CSP injection point, dev-mode exceptions, and WASM feasibility spike are
specified in `08.2a_phase_08_hwp_security_exact_diffs.md`.
Secure bridge module shapes and HWP event schema details are specified in
`08.2b_phase_08_hwp_security_revalidation_fixes.md`.

#### MODIFY React WebView HTML generation

Planned CSP policy:

```text
default-src 'none';
img-src ${webview.cspSource} data: blob:;
font-src ${webview.cspSource} data:;
style-src ${webview.cspSource} 'unsafe-inline';
script-src ${webview.cspSource};
frame-src ${webview.cspSource};
connect-src ${webview.cspSource};
worker-src ${webview.cspSource} blob:;
```

If rhwp WASM requires relaxed directives, the plan must state the exact reason.
Do not add broad `https:` or `*` to make loading pass.

#### MODIFY HWP message contract

Replace generic event names:

```text
init
hwpData
saveHwp
hwpSaved
```

With HWP-scoped messages:

```text
hwp:init
hwp:fileData
hwp:requestSave
hwp:saveResult
hwp:error
```

Every payload must be runtime checked before it crosses a trust boundary.
React-local load status is handled through the `onLoadStatus` bridge callback,
not through this extension-host message contract.

### Phase 8.2D — Decide Viewer-Only vs Real Editable

This is the main product decision. Do not mix the two modes.

Editable provider registration diffs are specified in
`08.2a_phase_08_hwp_security_exact_diffs.md`.
Provider registration and viewer safety diffs are further specified in
`08.2b_phase_08_hwp_security_revalidation_fixes.md`.

#### Option 1: Viewer-Only MVP

Use current `CustomReadonlyEditorProvider`.

Allowed:

```text
- Open .hwp/.hwpx
- Render pages
- Zoom/fit controls
- Copy/select if rhwp supports it safely
- Show "editing not yet supported"
```

Forbidden:

```text
- Save HWP
- claim editing support
- dirty state badges
- silent conversion writes
```

#### Option 2: Editable HWP Phase

Create a dedicated editable provider.

Required implementation:

```text
src/provider/hwp/HwpCustomDocument.ts
src/provider/hwp/HwpEditorProvider.ts
src/provider/hwp/hwpSaveService.ts
src/provider/hwp/hwpMessageSchema.ts
src/react/view/hwp/HwpEditor.tsx
```

Required VS Code lifecycle:

```text
CustomEditorProvider<HwpCustomDocument>
onDidChangeCustomDocument
saveCustomDocument
saveCustomDocumentAs
revertCustomDocument
backupCustomDocument
```

Hard blocker:

```text
`@rhwp/editor` currently exposes no document-changed event.
Editable mode requires either:
1. a patched local rhwp-studio that emits document dirty/change events, or
2. a conservative model that marks dirty after any user command and can still
   guarantee save/revert semantics.
```

Recommendation:

```text
Ship 8.2 as viewer-only first.
Create editable HWP as Phase 8.3 only after local bridge exposes change events.
```

### Phase 8.2E — File Safety

#### MODIFY `src/provider/handlers/hwpHandler.ts`

Exact constants, read migration, collision handling, and atomic write sequence
are specified in `08.2a_phase_08_hwp_security_exact_diffs.md`.

Required write behavior when save is re-enabled:

```text
1. Validate byte payload size and file signature before writing.
2. For `.hwp`, ask before overwriting original unless invoked through VS Code save lifecycle.
3. For `.hwpx`, default target must not be `basename.hwp` if it already exists.
4. Use `showSaveDialog` or generate `basename.converted.hwp`.
5. Write to a temp URI first, then rename/copy into place.
6. Emit a typed `hwp:saveResult` with the final URI and mode.
```
