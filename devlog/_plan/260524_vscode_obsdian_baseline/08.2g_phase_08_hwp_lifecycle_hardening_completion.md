# Phase 8.2g — HWP/HWPX Lifecycle Hardening Completion

Created: 2026-05-29
Status: completed implementation record
Parent: `08.2f_phase_08_hwp_full_editing_recovery_audit.md`

## Purpose

This document closes the hardening debt that remained after full rhwp editing
was restored. The product requirement remains unchanged:

```text
HWP/HWPX must expose the full rhwp editor, not a viewer-only wrapper.
```

The hardening strategy is therefore not to disable editing. Instead, HWP/HWPX is
now separated into a real editable VS Code custom editor and the full editor is
kept while save, backup, CSP, and message boundaries are tightened.

## Implemented Changes

### Editable Provider Split

```text
NEW  src/provider/hwp/HwpCustomDocument.ts
NEW  src/provider/hwp/HwpEditorProvider.ts
NEW  src/provider/hwp/hwpSaveService.ts
MOD  src/extension.ts
MOD  src/provider/officeViewerProvider.ts
MOD  package.json
```

`*.hwp` and `*.hwpx` are no longer owned by the readonly office viewer. They are
handled by `cweijan.hwpEditor`, which registers an editable
`CustomEditorProvider`. The package contribution marks this editor as the default
opener for both extensions.

Implemented lifecycle:

- `openCustomDocument`
- `resolveCustomEditor`
- `onDidChangeCustomDocument`
- `saveCustomDocument`
- `saveCustomDocumentAs`
- `revertCustomDocument`
- `backupCustomDocument`

Dirty state is driven by a conservative webview signal. This is intentionally a
content-change event, not an undo/redo edit stack, because current rhwp studio
does not expose a precise host-side edit event stream.

`src/extension.ts` also repairs stale user-level editor associations from the
previous implementation. If an existing profile still has `*.hwp` or `*.hwpx`
mapped to `cweijan.officeViewer`, startup migrates those entries to
`cweijan.hwpEditor` so upgraded users do not fall back to VS Code's binary text
editor.

### Save And Format Safety

```text
MOD  src/provider/handlers/hwpHandler.ts
MOD  src/common/hwpMessageSchema.ts
MOD  src/react/view/hwp/Hwp.tsx
```

Toolbar save remains available by default. VS Code native save now works through
an export request/response handshake:

```text
Provider -> hwp:vscodeSave(requestId, format)
React    -> exportHwp/exportHwpx according to requested format
React    -> hwp:vscodeSavePayload(requestId, bytes, format)
Provider -> validate magic, atomic write, clear dirty
```

Save As now exports according to the destination extension, not the source
extension. This closes the employee-audit finding where `doc.hwpx` could be
saved as `doc.hwp` while still containing ZIP/HWPX bytes.

Write guards:

- HWP bytes must match OLE magic;
- HWPX bytes must match ZIP magic;
- destination extension and exported format must match when destination is
  explicitly `.hwp` or `.hwpx`;
- writes use temp-file then rename;
- toolbar `.hwpx -> .hwp` conversion uses `.converted.hwp` and prompts on
  collision.

### Local Default And CSP

```text
MOD  src/common/reactApp.ts
MOD  src/react/util/vscodeConfig.ts
MOD  package.json
```

`vscode-obsdian.hwp.studioUrl` now defaults to an empty string. Empty means the
bundled local `resource/rhwp-studio` is used. A remote URL can still be supplied
explicitly, but CSP sources are derived from that configured URL instead of
hardcoding a broad remote allowance.

The final implementation mounts the local rhwp-studio bundle directly inside
the host WebView document. The earlier nested `srcdoc` iframe approach was
rejected because VS Code's WebView resource service worker could not resolve
nested `asWebviewUri` asset requests reliably. The direct mount preserves the
full rhwp editor surface while keeping the default runtime local.

The local bridge also rewrites rhwp's absolute `/assets/*.wasm` fetches back to
the bundled `resource/rhwp-studio/assets/*.wasm` WebView URI. This fixed the
observed Insiders runtime failure:

```text
Cannot read properties of undefined (reading '__wbindgen_malloc')
```

CSP now supports:

- dynamic `frame-src` origins;
- dynamic `connect-src` origins;
- local `vscode-webview:` rhwp-studio iframe;
- trusted remote `hwp.studioUrl` opt-in.
